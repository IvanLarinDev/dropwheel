# Dropwheel — заметка для нейросети

Оверлей-лаунчер для Windows 10/11: плавающий орб раскрывается в радиальное колесо целей
(папки и приложения); на плитки бросают файлы и текст, правила-сортировщики раскладывают
файлы по подпапкам.

Язык и стек: C#, .NET 10 (`net10.0-windows`), WPF. Тесты — xUnit. Решение —
`Dropwheel.slnx`; быстрый запуск — `run.cmd`.

## Карта проекта

- `src/Dropwheel/` — код приложения (точка входа `App.xaml.cs`). Слои: `Models/` (данные),
  `Services/` (логика без UI), `UI/` (WPF-окна). У каждой папки свои README/LLM.
- `tests/Dropwheel.Tests/` — xUnit; проект приложения открыт тестам через InternalsVisibleTo.
- `docs/demo/` — браузерная JS-демонстрация колеса (GitHub Pages); геометрия, тайминги и
  цвета зеркалят настоящий UI — меняешь `src/Dropwheel/UI`, не забудь про демо.
- `reports/`, `inbox/` — машинно-сгенерированные отчёты внутреннего инструментария;
  в архивы релизов не входят (`export-ignore`), руками не редактировать.

## Правила стиля

- Комментарии, docstring и сообщения логов в коде — только на английском (по-русски
  общаемся в чате и пишем документацию папок).
- Плоско и понятно: чистые функции где можно, KISS и DRY, без лишних слоёв абстракций.
- Разделение слоёв строгое: данные → `Models`, логика → `Services`, интерфейс → `UI`.
- Перед правкой и перед новой функцией ищи существующие использования и похожую логику
  через `ck` (лексический/семантический/гибридный поиск) — не дублируй.

## Ошибки

Не глотать: логировать через `Services/ErrorLog` или пробрасывать дальше. В нативных
callback-ах (низкоуровневые хуки мыши/клавиатуры) обязателен try/catch — исключение там
роняет процесс. Новые значения enum-ов из `config.json` добавляй в санитайзер
`TargetStore.TrySanitizeInvalidEnums`, иначе неизвестный токен сотрёт конфиг пользователя.

## Процесс и коммиты

- Коммиты — conventional commits (`feat:`, `fix:`, `docs:`…), описание короткое.
- `main` обновляется только через pull request; прямой push заблокирован.
- Изменения внешнего вида в `src/Dropwheel/UI/**` проходят дизайн-гейт (pre-push):
  нужны мокапы или waiver с пояснением, что визуал не менялся.
- Релиз: бамп версии на `main` → тег `v*` → публикация.
- История работы и принятые решения ведутся вне репозитория
  (`C:\Users\poweruser\.claude\projects\<имя-проекта>\`), в git их не тащить.

## Что нельзя

- Открывать окна и трогать WPF из `Services`; класть логику в code-behind `UI`.
- Глотать исключения пустым `catch`.
- Хардкодить цвета в UI мимо ролей `Palette`/`Theme`.
- Менять формат `config.json` без миграции и санитайзера.
- Коммитить в `main` напрямую или переписывать историю опубликованных веток.
