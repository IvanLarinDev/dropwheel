# scripts — заметка для нейросети

Роль папки: `release.ps1` — единственный вход конвейера релиза. Подробный
человеческий порядок действий — в `docs/RELEASING.md` и здешнем `README.md`.

## Ключевые факты о release.ps1

- Запуск: Windows PowerShell 5.1 (`pwsh` на машине нет, скрипт совместим с 5.1).
  Сначала всегда `-DryRun`. Вывод перенаправлять только stdout (`| Out-File`):
  `*>` в 5.1 оборачивает stderr git-а в ErrorRecord и валит скрипт.
- Строит из `origin/main` в изолированном worktree во временной папке — незапушенные
  коммиты в релиз не попадают. Меняет ровно два файла (allowlist): csproj и
  `CHANGELOG.md`.
- Порядок: release-commit → push в main → зелёный `ci.yml` → тег → `release.yml` →
  проверка GitHub-релиза (3 обязательных артефакта) → fast-forward локального main.
- Resume: повторный запуск с явной версией, равной уже забампленной на `main`
  (`-Bump 0.20.0`), находит существующий release-commit и доделывает недостающие
  шаги. `previousTag` при resume берётся из версии РОДИТЕЛЯ release-commit
  (`git show sha^:csproj`), а не из текущей — иначе resume требовал бы тег, который
  сам ещё не запушил.
- Сеть: `Invoke-NativeRetry` (3 попытки, 5 с) на всех `git fetch`/`git push` —
  против мимолётного «Empty reply from server». `Test-RemoteTag` различает
  «тега нет» (exit 2 у `ls-remote --exit-code`) и сетевой сбой (прочие коды →
  ретраи → throw): сбой не должен читаться как «тега нет», иначе скрипт пытается
  перепушить существующий тег.

## Локальные запреты

- Не ставить теги и не создавать GitHub-релизы мимо скрипта.
- Не ослаблять allowlist изменяемых файлов и проверку «тег только после зелёного CI».
- Синтаксис держать совместимым с PowerShell 5.1 (без `??`, тернарников и
  `&&`/`||`-цепочек).

Связи: пушит в `origin/main`, дергает `gh` (workflow `ci.yml`, `release.yml`).
Общие правила — в корневом `LLM.md`.
