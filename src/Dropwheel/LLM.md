# src/Dropwheel — заметка для нейросети

Роль папки: сам код приложения Dropwheel — оверлей-лаунчер для Windows (плавающий орб →
радиальное колесо целей; дроп файлов/текста, сортировщики папок).

Язык и стек: C#, .NET 10 (`net10.0-windows`), WPF. Тесты — xUnit в
`tests/Dropwheel.Tests` (проект открыт им через InternalsVisibleTo).

## Карта папки

- `Models/` — POCO-данные и enum-ы, формат `config.json`. Свои README/LLM внутри.
- `Services/` — вся логика без UI (файлы, запуск, хуки, вотчер, сортировка, иконки).
- `UI/` — WPF-окна; главное `OverlayWindow` разбито на `partial`-части по темам.
- Точка входа — `App.xaml.cs` (+ `App.Tray.cs`). Манифест — `app.manifest`.

## Правила стиля и ошибок

- Разделение слоёв: данные → `Models`, логика → `Services`, интерфейс → `UI`.
- Комментарии и docstring в коде — на английском (в чате общаемся по-русски). Плоско и
  понятно, без лишних абстракций; чистые функции где можно.
- Ошибки не глотать: логировать через `ErrorLog` или пробрасывать. В нативных callback-ах
  (хуки мыши/клавиатуры) обязательно try/catch — иначе падает процесс.
- Enum-ы в `config.json` строгие: новое значение добавляй и в санитайзер
  `TargetStore.TrySanitizeInvalidEnums`, иначе неизвестный токен сотрёт конфиг.

## Дизайн-гейт

Изменения внешнего вида/раскладки/анимаций в `UI/` требуют мокапов (хук `design-gate`,
жёсткий в pre-push/CI). Чисто логическая правка UI-файла допустима, но при пуше нужен
waiver с пояснением, что визуал не менялся.

## Процесс

Перед правкой ищи использования через `ck`, чтобы не сломать вызовы и не продублировать
логику. История работы и решения — вне репозитория (см. корневой `LLM.md`). Общие правила
проекта и формат коммитов — там же.
